# 診断

エディタ診断は `ss lsp` から送られます．言語サーバはプロジェクトのスナップショットを作り，`.ss` ファイルに対して診断を公開します．

## 診断の発生元

診断は parser，module loader，checker，展開処理，配置解決，ユーザ定義の report から発生します．

| 発生元 | 例 |
| --- | --- |
| parser | 不正な token，閉じていない文字列，不正な宣言 |
| module loader | 不明な `std:` import，見つからないプロジェクト import |
| 型検査 | 不明な関数，引数個数の不一致，引数型の不一致 |
| プロパティ検査 | 不明なフィールド，不正な値ドメイン |
| 展開 | 不正な文脈，生成処理の失敗 |
| 配置 | 制約衝突，object のはみ出し |
| ユーザ report | `report_error`，`report_warning` |

描画時だけに分かる失敗は，プレビュー描画を実行するまでエディタ診断にはなりません．`pdflatex` 不足，PDF 変換失敗，`qpdf` 失敗などが該当します．

## 更新

拡張は document open，change，save，close を言語サーバへ送ります．`*.ss` と `ss.toml` も監視します．

`ss: Check Current File` はアクティブな `.ss` ファイルを保存します．保存によって通常の言語サーバ更新が走ります．

## CLI との比較

エディタ診断の原因が分かりにくい場合は，プロジェクトルートで同じ CLI コマンドを実行します．

```sh
ss check --project .
```

エディタが別のバイナリを使っている場合は，`ss.cli.path` をターミナルと同じ実行ファイルに合わせます．

```json
{
  "ss.cli.path": "/path/to/ss/zig-out/bin/ss"
}
```

## プロジェクト解決

言語サーバは編集中のファイルからプロジェクト情報を解決します．`ss.toml` が見つからない場合は，アクティブなファイルを入口とし，そのファイルのディレクトリをアセット基準にします．

プロジェクト関連の診断では，次のパスを確認します．

| パス | 確認内容 |
| --- | --- |
| `ss.toml` | 想定したルートに存在するか |
| `entry` | `ss.toml` からの相対パスになっているか |
| `asset_base_dir` | ディレクトリが存在し，参照アセットを含むか |
| ローカル import | import された `.ss` ファイルが存在するか |

パス診断と描画コマンドの診断を分ける場合は，`ss doctor --project .` を実行します．

## 出力チャンネル

拡張は言語サーバとプレビューの出力を `ss` 出力チャンネルに書きます．エディタ診断とターミナル出力が違う場合に確認します．

```text
View -> Output -> ss
```
